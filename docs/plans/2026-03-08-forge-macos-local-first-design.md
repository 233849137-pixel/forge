# Forge MVP PRD（macOS 首发 / Local-First 单 App）

## Overview

Forge 是一个面向 AI 转型服务团队的内部研发中台，首发形态为 `macOS 桌面 App`。  
它不是纯 Web，也不是纯离线工具，而是 `单一 App 体验 + 本地执行 + 联网共享` 的混合产品。

Forge 的核心目标不是“造一个很大的平台”，而是把标准型 AI 项目的交付链路压缩成一条稳定流水线：

`需求确认 -> 模板继承 -> 资产复用 -> AI 生成 -> 本地构建测试 -> 预览交付 -> 经验沉淀`

在需求清晰、模板齐备的前提下，首版目标是让标准项目在 `3-5 天内产出内部可用版本`，并在 `0.5-1 个工作日内` 拉起新的同类项目。

## Product Definition

- 产品代号：`Forge`
- 首发平台：`macOS`
- 产品定位：内部 AI 研发中台桌面 App
- 核心用户：
  - 项目负责人
  - 研发工程师
  - 兼职外包
- 产品原则：
  - 只有一个 App，不在多个平台之间来回切换
  - 生产过程和项目资料默认本地执行、本地保存
  - 只有需要协作和复用的数据才同步到云端
  - 首版优先解决“启动快、复用强、质量可控”

## Goals

- 新项目启动时间 `< 30 分钟`
- 标准项目首个可演示版本产出时间 `< 1 工作日`
- 模板/Prompt/Skill 复用率 `> 60%`
- 交付前 P0/P1 缺陷清零
- AI 调用成本控制在项目收入的 `5%` 以内

## Product Shape

Forge 对用户表现为一个 `macOS 桌面 App`，但内部包含三层能力：

1. `桌面应用壳`
   - 统一提供项目列表、资产中心、Prompt 管理、任务状态、预览结果、测试报告。
   - 作为团队唯一入口。

2. `本地执行引擎`
   - 负责本地文件读写、Git 操作、Codex 调用、构建、Playwright 测试、日志采集。
   - 所有生产行为默认发生在本机。

3. `云同步服务`
   - 负责模板索引、共享资产、项目状态、成本数据、审计日志、团队协作信息。
   - 只同步必要的共享数据，不默认上传源代码和客户原始资料。

## Local vs Cloud Data Boundary

### 本地优先保存

- 项目代码
- 客户原始资料
- 知识库文件
- Prompt 调试草稿
- 构建缓存
- Playwright 录像与截图
- API 密钥与本机环境变量

### 云端共享保存

- 项目元信息
- 模板目录与版本
- Prompt/Skill 注册表
- 项目状态与负责人
- 成本统计
- 审计日志
- 归档后的可复用经验摘要

## In Scope

- 项目模板库
- Prompt 资产库
- Skill/模块复用清单
- 项目工作区
- 任务包生成器
- 本地构建与预览
- 基础质量门禁
- 项目驾驶舱
- 项目归档与经验沉淀

## Out of Scope

- iOS、Windows、Android 端
- 客户直接登录 Forge
- 可视化拖拽式多 Agent 编排器
- 完整 SaaS 多租户系统
- 自研模型平台
- 通用 Skill 市场
- 复杂 K8s 平台化部署

## Core Workflow

1. 项目负责人在 App 中创建项目，并选择行业模板与技术模板。
2. Forge 自动生成项目 DNA，包括技术栈、默认 Prompt、测试清单、部署模板、禁忌项。
3. 工程师在同一个 App 内查看项目工作区，选择可复用 Prompt、Skill 和代码模板。
4. 系统生成标准任务包，供 Codex 执行代码生成或改造。
5. 本地执行引擎自动完成构建、测试与预览。
6. 负责人在 App 内查看状态、质量结果和预览链接。
7. 交付完成后，系统将模板、Prompt、修复记录与经验摘要沉淀到共享资产。

## Functional Requirements

### FR-001: 项目创建
当项目负责人选择模板并点击创建项目时，系统应自动创建本地项目工作区，并注入默认文档、Prompt、测试清单和部署配置。

### FR-002: 项目 DNA 注入
当用户在项目内发起 AI 研发任务时，系统应自动附带项目 DNA、客户约束、禁忌项和推荐资产。

### FR-003: 资产检索
当用户按场景、行业或技术标签搜索时，系统应返回可复用的 Prompt、Skill、模板和测试包，并按最近成功记录排序。

### FR-004: 任务包生成
当用户点击“生成任务包”时，系统应输出一份可直接提供给 Codex 的标准任务包。

### FR-005: 本地构建
当代码生成完成或用户提交变更时，系统应自动执行本地构建，并生成可查看的构建状态和日志。

### FR-006: 本地预览
当构建成功时，系统应自动生成本地预览入口，并在 App 内展示可访问链接。

### FR-007: 质量门禁
当用户尝试标记项目为“可交付”时，系统应自动执行 lint、类型检查、基础测试和 Playwright 冒烟测试；任一失败时必须阻止交付。

### FR-008: 缺陷回流
当测试或门禁失败时，系统应自动生成修复任务，并附带失败日志、截图和失败阶段。

### FR-009: 项目驾驶舱
当用户进入项目列表或项目详情时，系统应展示状态、负责人、最近构建结果、成本和风险提示。

### FR-010: 经验归档
当项目被标记为已交付时，系统应将可复用模板、Prompt、修复摘要和测试摘要归档到共享资产中心。

### FR-011: 云端同步
当项目元信息、共享资产或审计数据发生变化时，系统应在联网状态下同步到云端。

### FR-012: 离线可用
当网络不可用时，系统应继续支持本地项目浏览、资料读取、任务生成、本地构建和测试；待恢复联网后再补同步。

## Non-Functional Requirements

### Performance

- 项目列表加载时间 `< 2s`
- 本地任务包生成时间 `< 10s`
- 标准构建完成并产出预览时间 `< 10 分钟`

### Security

- 本地密钥必须存储在 macOS Keychain 或等效安全存储中
- 项目间数据默认隔离
- 云端只接收被允许同步的元数据和共享资产

### Reliability

- 断网不影响本地核心流程
- 同步失败必须可重试
- 本地构建和测试日志必须可追溯

### Scale

- 首版支持 `1-5` 名团队成员协作
- 首版支持并行管理 `10` 个以内项目

## Acceptance Criteria

### AC-001: 创建智能客服项目
Given 用户选择“智能客服”模板  
When 用户点击创建项目  
Then 系统在 5 分钟内完成本地工作区创建，并注入默认 Prompt、测试清单和部署模板

### AC-002: 生成任务包
Given 项目已初始化完成  
When 用户点击“生成任务包”  
Then 系统生成一份包含需求摘要、约束、资产引用和验收标准的任务包

### AC-003: 本地构建预览
Given 工程师完成一次代码生成或修改  
When 用户触发构建  
Then 系统在 App 内展示构建状态、日志和本地预览入口

### AC-004: 门禁阻断
Given Playwright 冒烟测试失败  
When 用户尝试将项目标记为“可交付”  
Then 系统拒绝通过，并自动创建修复任务

### AC-005: 离线工作
Given 用户设备处于离线状态  
When 用户打开已有项目并继续开发  
Then 系统仍可读取资料、生成任务包、执行本地构建和测试

### AC-006: 交付后归档
Given 项目已被标记为已交付  
When 用户点击“归档沉淀”  
Then 系统把可复用模板、Prompt、修复摘要和测试摘要同步到共享资产中心

## Error Handling

| 错误场景 | 处理方式 |
|---|---|
| 模板不存在 | 阻止创建项目并提示模板缺失 |
| 资产检索为空 | 允许继续，但标记当前项目无推荐资产 |
| 构建失败 | 展示日志并生成修复任务 |
| 测试失败 | 阻止交付并记录失败阶段 |
| 网络不可用 | 自动切换为离线模式，待恢复后重试同步 |
| 同步冲突 | 保留本地版本，提示用户选择合并或覆盖 |

## Architecture Suggestion

- 桌面壳：`Tauri`
- 前端界面：`React`
- 本地元数据：`SQLite`
- 本地工作区：文件系统 + Git 仓库
- 本地执行：Node.js CLI / 本地任务进程
- 测试：`Playwright`
- 云端服务：轻量 API + PostgreSQL

首版目标不是搭一个“大平台”，而是尽快做出一个稳定可跑的 `macOS 本地优先单 App`。  
因此不建议首版拆成复杂的多后端体系，也不建议一开始上重量级 Agent 编排框架。

## Final Decision

Forge 首发版本应定义为：

`一个 macOS 桌面 App，联网但不依赖云端运行；资料和生产过程默认在本地执行，模板、状态和经验按需同步共享。`
